# 사진 촬영 예약 플랫폼 MVP 구현 요청

## 1. 역할

당신은 이 프로젝트의 시니어 Product Engineer이자 CTO 역할을 수행합니다.

단순히 화면 시안을 제안하거나 기획 문서를 작성하지 말고, 실제 실행 가능한 서비스를 코드로 구현해 주세요.

항상 다음 기준으로 판단하세요.

* 실제 사용자가 이용할 수 있는가
* 실제 서비스 운영에 적합한가
* 유지보수하기 쉬운가
* 이후 기능을 추가할 때 구조를 크게 변경하지 않아도 되는가
* 불필요한 기능을 구현하고 있지는 않은가

내가 제안한 구조가 사용자 경험이나 기술적으로 좋지 않다면 그대로 따르지 마세요.

문제점을 구체적으로 설명하고 더 나은 구조를 제안한 뒤 구현해 주세요.

단, 확정된 제품 원칙이나 MVP 범위를 임의로 바꾸지는 마세요.

---

## 2. 프로젝트 목표

원하는 스타일의 사진을 남기고 싶은 사용자와 촬영자를 연결하는 사진 촬영 예약 플랫폼의 실제 동작 가능한 모바일 웹 MVP를 구현합니다.

이번 작업은 단순한 UI 프로토타입이 아닙니다.

다음 기능이 실제 데이터와 연결되어 동작하는 서비스를 목표로 합니다.

* 회원가입 및 로그인
* 사용자 프로필
* 촬영자 등록
* 포트폴리오 등록
* 촬영 서비스 등록
* 촬영자 및 결과물 탐색
* 촬영 서비스 선택
* 날짜와 시간 선택
* 장소 선택
* 예약 생성
* 예약 내역 조회

실제 결제, 채팅, 자동 정산 등은 이번 MVP에서 제외합니다.

## 3. 서비스 소개

이 서비스는 원하는 스타일의 사진을 남기고 싶은 사람과 그 순간을 촬영해 줄 촬영자를 연결하는 사진 촬영 예약 플랫폼입니다.

현재 사용자가 사진 촬영을 예약하려 할 때 다음과 같은 문제가 있습니다.

* 전문 스냅 촬영은 가격이 부담스러운 경우가 많습니다.
* 친구나 지인에게 사진 촬영을 부탁하기는 부담스럽습니다.
* 오픈채팅이나 개인 거래는 상대방을 신뢰하기 어렵습니다.
* 촬영 결과가 원하는 스타일로 나올지 판단하기 어렵습니다.
* 가격과 포함 사항이 명확하지 않은 경우가 많습니다.
* 촬영자별 포트폴리오, 가격, 일정, 후기를 한곳에서 비교하기 어렵습니다.
* 일반적인 예약 도구는 존재하지만 사진 결과물을 중심으로 탐색하고 예약하는 경험은 부족합니다.

이 문제를 해결하여 사용자가 원하는 사진을 발견하고, 촬영자를 비교하고, 안심하고 예약할 수 있도록 합니다.

장기적으로는 사람들이 사진이 필요해지는 순간 가장 먼저 떠올리는 플랫폼이 되는 것이 목표입니다.

## 4. 초기 타겟 사용자

초기 핵심 사용자는 인스타그램과 이미지 기반 SNS 사용에 익숙한 20~30대 여성입니다.

주요 촬영 목적은 다음과 같습니다.

* 자연스러운 일상 스냅
* 여행 사진
* 데이트 사진
* 커플 사진
* 친구와의 우정 사진
* 생일 및 기념일 사진
* 프로필 사진
* SNS 피드용 사진
* 반려동물 사진

인플루언서와 전문 크리에이터도 사용할 수 있지만, 장기적으로는 일반 사용자가 일상적인 촬영 수요가 생길 때 반복적으로 이용하는 플랫폼을 지향합니다.

## 5. MVP에서 검증할 가설

이번 MVP를 통해 다음을 확인할 수 있어야 합니다.

1. 사용자가 사진 결과물을 보고 촬영자를 탐색하는가
2. 사용자가 원하는 촬영 서비스를 선택하고 예약 단계까지 이동하는가
3. 가격, 지역, 결과물, 일정, 포함 사항 등의 정보가 예약 판단에 충분한가
4. 촬영 예약 과정이 복잡하거나 불안하게 느껴지지 않는가
5. 촬영자가 자신의 서비스와 가격을 직접 등록할 수 있는가
6. 사용자와 촬영자의 예약 정보가 올바르게 저장되고 조회되는가
7. 포트폴리오, 후기, 응답 상태 같은 신뢰 정보가 사용자가 개인 간 거래 대신 이 플랫폼을 통해 예약하도록 만드는가
8. 촬영자가 자유롭게 구성한 서비스(고정 등급 없음)를 사용자가 비교하고 이해하는 데 어려움이 없는가

## 6. 핵심 제품 원칙

### 6.1 사진이 탐색의 출발점이다

사용자는 촬영자 이름보다 결과 사진을 먼저 보고 관심을 가질 가능성이 높습니다.

사진을 가장 중요한 콘텐츠로 다루되, 예약 판단에 필요한 가격, 지역, 시간, 후기 등의 정보도 빠르게 확인할 수 있어야 합니다.

### 6.2 감성과 예약 정보가 함께 제공되어야 한다

사진 감상에만 집중하면 예약 조건을 파악하기 어렵고, 예약 정보만 강조하면 서비스의 매력이 사라집니다.

사진을 발견하는 경험과 실제 예약하는 경험이 자연스럽게 이어지도록 설계해 주세요.

### 6.3 가격과 포함 사항은 명확해야 한다

촬영 서비스의 가격, 촬영 시간, 결과물, 추가 비용 발생 조건을 예약 전에 명확히 확인할 수 있어야 합니다.

### 6.4 촬영자에게 상품 구조를 강제하지 않는다

플랫폼은 `라이트`, `베이직`, `시그니처` 같은 공통 상품명이나 가격 구성을 강제하지 않습니다.

촬영자는 자신의 촬영 서비스에 맞게 서비스명, 가격, 촬영 시간, 제공 결과물을 자유롭게 등록할 수 있어야 합니다.

### 6.5 사용자는 가입하지 않아도 둘러볼 수 있다

앱에 처음 진입하자마자 로그인을 요구하지 마세요.

홈, 탐색, 촬영자 상세는 로그인하지 않은 상태에서도 확인할 수 있어야 합니다.

찜, 예약, 촬영자 등록 등 사용자 데이터 저장이 필요한 시점에 로그인을 자연스럽게 유도합니다.

### 6.6 사용자와 촬영자는 별도 계정이 아니다

모든 회원은 일반 사용자로 가입합니다.

일반 사용자가 마이페이지에서 `촬영자로 활동하기`를 선택하면 촬영자 프로필을 생성할 수 있어야 합니다.

한 사람은 사진을 예약하는 사용자이면서 동시에 촬영자로 활동할 수 있습니다.

## 7. 기술 스택

다음 기술 스택을 사용해 주세요.

### 7.1 애플리케이션

* Next.js
* App Router
* TypeScript
* React Server Components를 기본으로 사용
* 클라이언트 상호작용이 필요한 부분에만 Client Component 사용

### 7.2 스타일링

* Tailwind CSS
* shadcn/ui는 필요한 컴포넌트만 선택적으로 사용
* Lucide React 아이콘 사용

shadcn/ui의 기본 디자인을 그대로 조합하여 전형적인 관리자 화면처럼 만들지 마세요.

서비스의 디자인 원칙에 맞게 조정해 사용해 주세요.

### 7.3 상태 관리

* 서버 데이터는 가능하면 서버에서 가져옵니다.
* 예약 단계처럼 여러 화면에서 임시 선택값을 유지해야 할 경우 Zustand를 사용합니다.
* 불필요한 전역 상태를 만들지 마세요.
* 필요한 경우 예약 진행 상태를 localStorage에 임시 저장합니다.

### 7.4 데이터 검증

* Zod
* 클라이언트 입력과 서버 요청을 모두 검증
* TypeScript의 `any`로 타입 오류를 회피하지 않음

### 7.5 백엔드

* Next.js Route Handlers
* 필요한 경우 Server Actions
* 별도의 Express 서버 또는 Spring Boot 서버는 사용하지 않음

중요한 데이터 검증과 권한 확인은 브라우저가 아니라 서버에서 처리하세요.

### 7.6 데이터베이스 및 인프라

* Supabase PostgreSQL
* Supabase Auth
* Supabase Storage
* Vercel 배포

### 7.7 추후 도입

* Toss Payments
* Sentry
* PostHog 또는 Google Analytics

이번 초기 구현에서 결제와 분석 도구는 연결하지 않습니다.

## 8. Supabase 사용 원칙

Supabase는 다음 용도로 사용합니다.

* PostgreSQL 데이터베이스
* 이메일 회원가입 및 로그인
* 사용자 세션
* 포트폴리오 및 프로필 이미지 저장
* Row Level Security
* 사용자별 데이터 접근 제어

### 8.1 보안 원칙

* 서비스 역할 키를 클라이언트에 노출하지 마세요.
* 클라이언트에서 전달한 사용자 ID를 그대로 신뢰하지 마세요.
* 인증된 세션의 사용자 ID를 서버에서 확인하세요.
* 모든 주요 테이블에 적절한 RLS 정책을 적용하세요.
* 공개 데이터와 개인 데이터를 구분하세요.
* `.env.local`을 Git에 커밋하지 마세요.
* `.env.example`에는 실제 키를 넣지 마세요.

### 8.2 Supabase 연결 전

환경변수가 아직 제공되지 않았다면 임의의 키를 만들거나 연결된 것처럼 구현하지 마세요.

다음 환경변수를 사용하는 구조만 준비하세요.

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
```

`SUPABASE_SERVICE_ROLE_KEY`는 정말 필요한 서버 작업에만 사용하고, 초기 구현에서는 사용을 최소화하세요.

## 9. 인증 및 회원가입

Supabase Auth를 사용합니다.

### 9.1 이번 MVP에서 실제 구현할 방식

* 이메일 회원가입
* 이메일 로그인
* 로그아웃
* 로그인 세션 유지
* 로그인한 사용자의 프로필 생성 및 조회

회원가입과 로그인 흐름은 실제로 동작해야 합니다.

### 9.2 후순위 로그인 방식

다음 방식은 이번 초기 구현에서 실제 연동하지 않습니다.

* Google 로그인
* Kakao 로그인

하지만 이후 쉽게 추가할 수 있도록 인증 컴포넌트와 인증 서비스 구조를 특정 로그인 방식에 종속되지 않게 설계해 주세요.

동작하지 않는 소셜 로그인 버튼을 실제 기능인 것처럼 노출하지 마세요.

소셜 로그인 버튼 UI가 필요하다면 `준비 중` 상태를 명확하게 표시하거나 실제 연동 시점에 추가합니다.

### 9.3 로그인 유도 시점

다음 화면은 비로그인 상태에서도 접근할 수 있어야 합니다.

* 홈
* 탐색
* 촬영자 목록
* 촬영자 상세
* 포트폴리오 확인

다음 행동을 할 때 로그인을 요구합니다.

* 촬영자 찜
* 예약 진행
* 내 예약 조회
* 촬영자로 활동하기
* 프로필 수정
* 후기 작성

### 9.4 사용자 프로필

Supabase Auth 사용자와 별도의 공개 프로필 테이블을 구성합니다.

예시 필드:

* id
* nickname
* avatar_url
* phone
* created_at
* updated_at

Auth 사용자 메타데이터만으로 서비스 프로필 전체를 관리하지 마세요.

## 10. 사용자와 촬영자 구조

사용자를 `customer`와 `photographer` 중 하나로만 구분하지 마세요.

한 사용자가 두 역할을 동시에 수행할 수 있어야 합니다.

권장 구조:

```text
auth.users
    │
    └── profiles
            │
            └── photographer_profiles
```

`photographer_profiles` 레코드가 없는 사용자는 일반 사용자입니다.

`photographer_profiles` 레코드를 생성하면 촬영자 등록 절차를 진행할 수 있습니다.

### 10.1 촬영자 상태

예시:

```ts
type PhotographerStatus =
  | "draft"
  | "pending_review"
  | "active"
  | "suspended";
```

MVP에서 실제 관리자 심사를 구현하지 않는다면 `draft`와 `active` 중심으로 단순화할 수 있습니다.

관리자 심사를 구현한 것처럼 가짜 승인 절차를 만들지 마세요.

### 10.2 모드 전환

촬영자 등록 전에는 마이페이지에 다음 진입점을 제공합니다.

```text
촬영자로 활동해 보세요
포트폴리오와 촬영 서비스를 등록하고 예약을 받아보세요.

[촬영자 등록 시작]
```

촬영자 프로필이 존재하는 사용자에게는 다음 전환 기능을 제공합니다.

* 사용자 모드
* 촬영자 모드

사용자 모드와 촬영자 모드는 별도 계정을 사용하지 않습니다.

### 10.3 내비게이션 예시

사용자 모드:

* 홈
* 탐색
* 내 예약
* 마이

촬영자 모드:

* 대시보드
* 예약
* 일정
* 프로필

이번 MVP에서는 촬영자 모드의 복잡한 대시보드와 통계를 구현하지 마세요.

## 11. 촬영 서비스 구조

플랫폼에서 `라이트`, `베이직`, `시그니처` 등의 촬영 상품명을 강제하지 않습니다.

촬영자는 자신만의 촬영 서비스를 자유롭게 등록합니다.

예시:

* 서울숲 자연광 감성 스냅
* 성수 데이트 스냅
* 야간 필름 무드 촬영
* 반려동물 산책 스냅
* 여행 동행 촬영
* 프로필 촬영

### 11.1 필수 정보

각 촬영 서비스는 다음 정보를 포함합니다.

* 서비스명
* 가격
* 촬영 시간
* 서비스 설명
* 포함 사항
* 활동 가능 지역
* 대표 이미지
* 공개 여부

### 11.2 선택 정보

* 보정본 수

* 필름

* 빈티지
* 청량한
* 도시적인

MVP에서 촬영자가 임의의 태그를 무제한 생성하게 하지 마세요.

## 12. 예약 시간 구조

### 12.1 핵심 원칙

촬영 서비스의 촬영 시간과 예약 시작 시간 단위를 분리합니다.

* 촬영 서비스의 시간은 자유롭게 설정
* 예약 가능한 시작 시간은 MVP에서는 기본적으로 30분 단위로 제공합니다.

예를 들어 촬영 서비스 시간은 다음처럼 다양할 수 있습니다.

* 30분
* 45분
* 60분
* 75분
* 90분
* 120분

예약 시작 시간은 다음처럼 표시합니다.

* 13:00
* 13:30
* 14:00
* 14:30

### 12.2 시간 점유 방식

서비스 시간이 75분이고 사용자가 13:00에 예약한다면, 예약 종료 시간은 14:15입니다.

다른 예약은 해당 촬영자의 준비 시간 정책을 고려하여 충돌하지 않는 시간부터 가능해야 합니다.

단순히 다음 세 개의 30분 슬롯만 문자열로 저장하는 방식보다 다음 정보를 기준으로 예약을 관리하세요.

* starts_at
* ends_at
* duration_minutes
* buffer_minutes

예약 충돌은 시간 구간이 겹치는지 서버 또는 데이터베이스에서 검증해야 합니다.

프론트엔드 버튼 비활성화만으로 중복 예약을 방지하지 마세요.

### 12.3 촬영 시간 입력

촬영자는 촬영 시간을 분 단위로 입력할 수 있어야 합니다.

MVP에서는 지나치게 자유로운 숫자 입력으로 오류가 생기지 않도록 다음과 같은 선택지를 제공해도 됩니다.

* 30분
* 45분
* 60분
* 75분
* 90분
* 120분
* 직접 입력

직접 입력 시 최소·최대 범위와 입력 단위를 검증하세요.



### 12.4 준비 및 이동 시간

데이터 구조에는 촬영 전후 여유 시간을 고려할 수 있는 필드를 준비하세요.

예시:

* buffer_before_minutes
* buffer_after_minutes

MVP UI에서는 단순화를 위해 촬영 후 여유 시간만 설정하게 해도 됩니다.

단순히 위치 기반 서비스처럼 보이게 하기 위해 지도 UI를 추가하지 마세요.

## 17. 디자인 원칙
이 서비스는 사진 감상 경험과 예약 경험이 자연스럽게 이어지는 모바일 웹 서비스입니다.

전체적으로 깨끗하고 신뢰감 있으며
감성적인 분위기를 유지하세요.
사진 자체가 서비스의 분위기를 만들도록 설계하세요.

### 17.1 핵심 경험

* 사용자가 앱을 열었을 때 사진을 둘러보고 싶어야 합니다.
* UI는 사진을 가리거나 경쟁하지 않아야 합니다.
* 예약에 필요한 가격, 지역, 시간, 후기, 포함 사항은 쉽게 확인할 수 있어야 합니다.
* 사진 감상에서 예약 행동으로 자연스럽게 이어져야 합니다.
* 감성적이지만 신뢰할 수 있는 인상을 줘야 합니다.

### 17.2 시각 원칙

* 충분한 여백
* 밝고 따뜻한 뉴트럴 컬러
* 명확한 타이포그래피 위계
* 사진 중심 레이아웃
* 심플한 버튼과 카드
* 일관된 이미지 비율
* 명확한 선택 상태
* 과하지 않은 테두리와 그림자

### 17.3 사용하지 않을 스타일

* 과도한 그라데이션
* 네온 컬러
* 글래스모피즘
* 과도하게 둥근 카드
* 모든 콘텐츠를 카드 안에 넣는 구조
* 불필요하게 많은 그림자
* 지나친 애니메이션
* AI 서비스에서 흔히 사용하는 보라색 중심 UI
* 실사용성이 낮은 Dribbble 스타일

### 17.4 모바일 기준

* 기준 너비: 375~430px
* 데스크톱: 최대 480px 모바일 컨테이너 중앙 정렬
* 주요 터치 요소는 44×44px 이상의 터치 영역을 권장하되, 모든 요소에 일괄 적용하지 않는다.
* 하단 고정 CTA 사용 시 하단 내비게이션, 콘텐츠 및 기기 안전 영역과 겹치지 않게 처리한다.
* 안전 영역 고려
* 모바일 브라우저 높이 변화 고려
* 페이지 전체에서 발생하는 의도하지 않은 가로 스크롤은 금지한다. 하지만 필터 칩, 카테고리 탭, 이미지 목록 등 가로 탐색이 적합한 컴포넌트는 해당 영역 내부에서 의도적인 가로 스크롤을 허용한다.
* 긴 텍스트 줄바꿈 처리
* 키보드가 입력 영역을 가리지 않게 처리
* 모바일 브라우저 주소창 변화에 대응할 수 있도록 고정 100vh 사용을 지양하고 dvh 등을 고려한다.
