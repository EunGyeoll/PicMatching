## 1. 프로젝트 셋업

| 결정 | 내용 |
|---|---|
| 프레임워크 버전 | Next.js 16.2.10 (App Router), React 19.2, TypeScript, Tailwind CSS v4 |
| 디렉터리 구조 | `src/` 디렉터리 사용, import alias `@/*` → `./src/*` |

## 3. Supabase 인프라

- 최초 프로젝트를 시드니(`ap-southeast-2`) 리전으로 생성했다가, 서울(`ap-northeast-2`)로 재생성함. Supabase는 프로젝트 생성 후 리전을 변경할 수 없어(인프라 레벨 고정), 데이터가 없는 상태였기에 새로 만들고 기존 프로젝트는 삭제하는 방식으로 처리.
- 최종 프로젝트: `PicMatching` (id: `ryxivaplpsbbjpfqryub`, 리전: `ap-northeast-2`)
- 환경변수 네이밍은 명세 8.2를 그대로 따름: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`(값은 Supabase의 최신 `sb_publishable_...` 키 사용), `SUPABASE_SERVICE_ROLE_KEY`.
- `service_role` 키는 MCP로 조회하지 않고, 사용자가 대시보드에서 직접 `.env.local`에 붙여넣는 방식으로만 다룸 (채팅에 노출 금지).

## 4. 데이터베이스 스키마

명세 18번 엔티티를 기준으로 5개 마이그레이션(`core_schema`, `functions_and_triggers`, `rls_policies`, `seed_service_tags`, `security_hardening`)으로 적용.

### 구조 결정
- `photographer_profiles.id`가 `profiles.id`를 그대로 공유(1:1 PK 공유) — "사용자이자 촬영자"(6.6)를 별도 `photographer_id` 개념 없이 표현.
- `shooting_services`/`bookings`에 `buffer_before_minutes`도 추가 (명세 12.4가 "촬영 후 여유시간만 UI에 노출해도 되지만, 데이터 구조에는 전후 필드를 모두 준비"하라고 명시).
- 서비스별 "활동 가능 지역"은 별도 정규화 테이블 없이 `shooting_services.areas text[]`로 단순화 (명세 18번 엔티티 목록에 `photographer_areas`만 있고 서비스 단위 지역 테이블은 없었음). 촬영자 단위 지역은 명세대로 `photographer_areas` 테이블 유지.
- 태그(촬영 목적/무드)는 `service_tags` 고정 테이블 + `photographer_service_tags` 조인 테이블로 구현, 자유 생성 불가 (11.3).
- 예약 시간 겹침 방지: PostgreSQL `EXCLUDE USING gist` 제약을 프론트/서버 검증과 별도로 DB 레벨에 추가. `starts_at`/`ends_at`에 버퍼 시간을 반영한 `blocked_range` 컬럼을 트리거로 미리 계산해 저장하고, 그 컬럼에 제약을 걺 (직접 계산식을 인덱스에 쓰면 PostgreSQL이 "IMMUTABLE 아님" 에러를 냄).
- 예약 생성 시 서비스명/가격을 스냅샷 컬럼(`service_title_snapshot`, `base_price_snapshot`, `total_price_snapshot`)에 저장 (18.2, 이후 서비스 가격이 바뀌어도 기존 예약 금액 불변).
- 후기(`reviews`)는 `booking_id`에 1:1 유니크로 연결 — 완료된 예약 없이는 후기 작성 불가 (16번 "실제 후기 수"의 신뢰성 보장).
- `booking_status_logs`는 클라이언트가 직접 insert할 수 없고, `bookings` insert/update 트리거가 `SECURITY DEFINER`로만 기록.

### RLS 정책 요지
- 전 테이블 RLS 활성화.
- `profiles`는 본인만 조회/수정. 닉네임/아바타만 공개하는 `profiles_public` 뷰를 별도로 만들어 전체 공개 (phone 등 민감 필드는 뷰에서 제외).
- `photographer_profiles`/`shooting_services`/`portfolio_images` 등은 `status='active'` 또는 `is_published=true`인 것만 공개 조회, 쓰기는 소유자만.
- `blocked_times`는 **촬영자 본인만 조회 가능** (공개 안 함) — 손님에게 보여줄 "예약 가능 시간"은 차단 사유 원본을 그대로 노출하지 않고 나중에 서버 함수로 계산해서 제공할 예정 (미구현, 다음 단계 과제).
- `service_tags`는 전체 공개 조회, 쓰기는 없음(플랫폼 관리 전용).

### 보안 하드닝
- 트리거 함수(`set_updated_at`, `bookings_set_blocked_range`)에 `search_path` 명시 (search_path 인젝션 방지).
- 트리거 전용 함수(`handle_new_user`, `log_booking_created`, `log_booking_status_change`)의 `EXECUTE` 권한을 `anon`/`authenticated`에서 회수 (API로 직접 호출 불가능하게 차단, 트리거 동작에는 영향 없음).

### 보류한 항목 (낮은 위험으로 판단, 추후 정리)
- `profiles_public` 뷰가 Supabase 보안 어드바이저에서 "SECURITY DEFINER View" ERROR로 표시됨 — 의도된 설계(공개 정보만 담은 뷰가 `profiles`의 RLS를 우회해 전체 공개되도록 함)라 그대로 둠.
- `btree_gist` 확장이 `public` 스키마에 설치되어 있음 (권장은 별도 스키마) — 예약 겹침 제약이 이 확장에 의존하고 있어, 옮기려면 제약을 재생성해야 함. 실사용 위험이 낮아 보류.

## 5. 인증 구현

- Server Actions + React `useActionState` 패턴 사용 (react-hook-form 등 추가 폼 라이브러리 도입하지 않음).
- Zod로 이메일/비밀번호/닉네임/연락처를 클라이언트·서버 양쪽에서 검증. 비밀번호 최소 길이는 Supabase 프로젝트 기본값(6자)에 맞춤.
- Supabase Auth 에러 코드를 자연스러운 한국어 메시지로 매핑하는 헬퍼(`toKoreanAuthError`)를 만들어 기술적 오류 원문이 그대로 노출되지 않게 함.
- **이메일 인증 요구 여부는 앱 코드가 아니라 Supabase 프로젝트 설정(`Confirm email`)에 위임.** 현재는 켜져 있는 기본값을 유지 중이며, 언제든 대시보드에서 끌 수 있음. 회원가입 액션은 두 경우(즉시 세션 발급 / 이메일 확인 필요) 모두 올바르게 동작하도록 구현.
- `/mypage`(허브)는 비로그인 시 인라인 안내 화면을 보여주고, `/mypage/edit`(수정)은 비로그인 시 서버에서 `/login`으로 리다이렉트 — 모두 클라이언트 숨김이 아니라 서버 단에서 처리.
- `proxy.ts`는 세션 쿠키 갱신만 담당하고, 페이지 단위 접근 제어(로그인 필요 여부)는 각 페이지에서 개별 처리 — 이 서비스는 비로그인 상태로 둘러볼 수 있는 화면이 많아(6.5, 9.3) 미들웨어 레벨의 일괄 리다이렉트를 쓰지 않기로 함.

## 8. 촬영자 온보딩 화면 구조

코드 작성 전 와이어프레임으로 먼저 검토·확정.

- **등록을 3단계로 분리**: 1) 활동명·프로필 이미지·한 줄 소개 → 2) 상세 소개·주요 활동 지역 → 3) 포트폴리오·연락처. 8개 필드를 필드 단위가 아니라 논리적 묶음 단위로 나눠, 명세 14.13 "단계를 지나치게 많이 나누지 않되 부담 없이 입력"을 만족시킴.
- **포트폴리오 업로드 그리드는 온보딩 3단계와 `/photographer/portfolio`(상시 관리 화면)가 동일 컴포넌트를 재사용.**
- **포트폴리오 최소 3장을 온보딩에서 강제.** 사진 없는 촬영자가 탐색 화면에 노출되는 걸 막기 위함 (6.1 "사진이 탐색의 출발점" 원칙).
- **등록 완료 시 상태는 바로 `active`.** 10.1에 따라 이번 MVP는 관리자 심사가 없으므로 `pending_review` 같은 가짜 대기 상태를 거치지 않음.
- **촬영자 프로필 단위의 "촬영 스타일" 구조화 필드는 만들지 않기로 함.** 명세 14.13은 등록 화면에서 촬영 스타일(무드) 선택을 요구하지만, 무드 태그 테이블(`photographer_service_tags`)은 서비스 단위로만 연결되도록 이미 설계되어 있음(4번 항목). 촬영자 프로필에 별도 태그 테이블(`photographer_styles` 등)을 추가하면 "촬영자가 스스로 고른 스타일"과 "실제 서비스에 붙인 태그"가 서로 어긋날 수 있어, 새 테이블을 만들지 않고 등록 화면에서는 상세 소개(자유 서술)로만 받기로 함. 구조화된 무드 태그 선택은 서비스 등록 화면에서만 이뤄지며, 촬영자의 "스타일"은 결과적으로 그가 등록한 서비스 태그의 합으로 드러남.
- **서비스 등록은 단계가 아니라 아코디언으로 분리.** 필수 8개(11.1) + 선택 9개(11.2) 총 17개 필드를 반복 등록해야 하는 화면이라 단계 이동 비용이 큼 — 필수 필드는 항상 펼쳐두고, 선택 필드(보정본 수·원본 제공·전달 기간·최대 인원·추가 비용 조건 등)만 접힌 아코디언으로 그룹핑.

## 9. 작업 순서 재조정 + 시드 데이터

- **촬영자 온보딩 UI(3단계 화면, 포트폴리오 관리, 서비스 등록, photographer 레이아웃) 구현을 뒤로 미룸.** 명세 5번 "검증할 가설"이 대부분 고객 쪽 탐색·예약 경험에 관한 것이라, 실제 데이터를 SQL로 빠르게 채워 넣고 홈/탐색/촬영자 상세/예약 플로우(고객 핵심 루프)를 먼저 구현하기로 함. 이미 만든 지원 파일(Storage 버킷, `complete_photographer_onboarding` RPC, `ImageUploader`, `TagInput`, Zod 스키마)은 그대로 두고 나중에 이어서 화면만 붙이면 됨.
- **테스트 촬영자 계정(`test-photographer@example.com`)에 프로필 1개 + 활동지역 2개 + 포트폴리오 5장 + 촬영 서비스 2개 + 태그를 SQL로 직접 시드.** 값은 임의로 채움(실제 사용자 데이터 아님).
- **포트폴리오/서비스 대표 이미지는 외부에서 다운로드하지 않고 로컬에서 생성한 단색 PNG를 사용.** 이 개발 환경은 Supabase API 도메인 외 일반 인터넷(예: picsum.photos)에 접근할 수 없어서, Node로 최소한의 유효한 PNG 파일을 직접 만들어 Storage에 업로드함 — 실제 스토리지 경로·공개 URL이 정상 동작하는지까지 확인하기 위해 가짜 경로 문자열 대신 실제 파일을 넣음. 나중에 실제 프로필 이미지로 교체하면 됨.
- anon 키로 `photographer_profiles`, `shooting_services`, 포트폴리오 이미지 공개 URL을 직접 조회해 RLS와 Storage 공개 접근이 실제로 동작하는 것을 확인함.

## 10. 검증 가설 보강

`MVP_IMPLEMENTATION_SPEC.md` 5번 섹션에 7, 8번 가설을 추가함.

- 기존 6개는 탐색→선택→예약 퍼널과 기술적 정합성 위주였고, 3번(서비스 소개)이 내세운 핵심 차별점("개인 거래는 신뢰하기 어렵다")과 6.4 원칙("공통 상품 등급을 강제하지 않는다")이 검증 대상에서 빠져 있어 추가함.
- 7. 포트폴리오·후기·응답 상태 같은 신뢰 정보가 사용자가 개인 간 거래 대신 이 플랫폼을 통해 예약하도록 만드는가
- 8. 촬영자가 자유롭게 구성한 서비스(고정 등급 없음)를 사용자가 비교하고 이해하는 데 어려움이 없는가
- 새 기능 구현이 아니라 출시 후 관찰 지표를 늘리는 것이라 MVP 범위에는 영향 없음.

## 11. 탐색 경험 화면 구조

코드 작성 전 와이어프레임으로 검토·확정 (2차: 홈 구조는 세 번 수정을 거쳐 확정).

- **`/explore`와 `/photographers`를 통합.** 별도 화면 대신 탐색 화면 안에서 **[사진]/[촬영자] 보기 토글**로 전환하는 구조로 결정 — 두 화면이 검색·필터 UI를 그대로 공유하고 결과 카드 형태만 다르기 때문. 명세 14.3도 이 방향을 검토하라고 명시했음. `/photographers`는 `/explore?view=photographers`로 리다이렉트만 남김. (11번 항목에 있던 열린 질문 해소)
- **홈 화면은 큐레이션 가로 스크롤 섹션 구조로 확정.** 처음엔 세로 사진 피드(인스타 익스플로어 탭 방식)를 검토했으나, 초기 단계는 촬영자 수가 적어 세로 피드가 몇 장 만에 끊겨 콘텐츠 부족이 더 도드라져 보이는 문제가 있어 기각. 가로 스크롤 섹션은 카드 몇 개만 있어도 자연스러워 보여 초기 마켓플레이스에 유리함. 카드를 한 차례 208px(사진 위 텍스트 오버레이)로 키웠다가, 화면에 비해 크다는 피드백으로 원래 크기(168px, 사진 아래 별도 텍스트)로 되돌림 — 최종 카드 크기는 168px.
- **위치·검색은 풀 너비 검색바 대신 작은 pill + 아이콘 한 줄로 축소.** 사진 콘텐츠 앞에 텍스트 UI가 차지하는 공간을 최소화.
- **"인기 촬영 지역" 홈 섹션은 만들지 않음.** 탐색 화면의 지역 필터로 흡수 — 첫 화면 섹션 수를 줄이기 위함(14.1 "지나치게 많은 섹션 금지").
- **무드는 칩 줄이 아니라 "무드로 찾기" 시각 섹션(사진+라벨 그리드)으로 확정.** 한 차례 칩 줄로 압축했다가, 그러면 화면 하단에 콘텐츠가 없어 보인다는 피드백으로 시각 섹션으로 복귀. 최종 홈 구조: 위치/검색 한 줄 → 촬영목적 칩 1줄 → "지금 만나볼 수 있는 촬영" 섹션 → "무드로 찾기" 섹션.
- **촬영자 상세 화면은 인스타그램 프로필 패턴 적용.** 원형 프로필 사진(`profiles.avatar_url`) + 이름 + 소개를 가운데 정렬한 헤더, 그 아래 포트폴리오/등록서비스/완료촬영/후기 통계 행, **[포트폴리오]/[촬영 서비스] 탭 전환**(포트폴리오는 3열 정사각 그리드로 감상 모드, 촬영 서비스는 리스트로 예약 판단 모드). 이 화면은 이미 데이터가 확정된 화면이라 "콘텐츠 부족" 문제가 없어 인스타식 구조를 그대로 적용해도 무방하다고 판단 — 홈과는 다른 결론.
- **신뢰 정보는 실제 값만 표시.** 후기 0건은 "후기 없음"으로, 완료 촬영은 실제 숫자(0 포함)로. 응답 상태 배지는 응답시간 추적 기능이 없어 아예 넣지 않음.
- **"가능한 일정 요약"(14.4)은 지금 비워둠.** 시드된 촬영자가 아직 `availability_rules`를 등록하지 않아 보여줄 데이터가 없음 — 예약 플로우 와이어프레임에서 이어서 다룸.

## 12. 예약 플로우 화면 구조 + 추가 시드

코드 작성 전 와이어프레임으로 검토·확정. 서비스선택→일정→장소→예약확인 4단계는 상단 진행 표시를 공유.

- **서비스 선택 화면(14.5)은 라디오 선택형.** 촬영자 상세에서 특정 서비스 카드로 진입하면 해당 서비스가 미리 선택된 채로 열리고, 선택된 카드만 전체 필드(보정본수/원본제공/전달기간/최대인원/추가비용)를 펼쳐서 보여줌 — 두 서비스를 동시에 다 펼치면 정보 과잉이라 판단.
- **장소 선택은 4개 탭(촬영자 추천/인기 장소/직접 입력/추후 협의)**, `locations` 테이블이 비어 있어도 "직접 입력"·"추후 협의" 탭은 데이터 없이 항상 동작.
- **예약 확인 화면의 금액은 클라이언트 값을 신뢰하지 않고 서버가 재조회.** 22번 원칙 그대로.
- **예약 완료 화면은 "요청 완료"이지 확정이 아님을 문구로 명시.**
- **`availability_rules`(요일별 운영시간)를 테스트 촬영자에게 시드함.** 일정 선택 화면이 빈 상태만 뜨는 문제를 막기 위해, 화~일요일 12:00–20:00(월요일 휴무)으로 등록 — 실제 예약 생성까지 끝까지 검증하려면 필요했음. `locations`는 "직접 입력" 탭이 데이터 없이도 동작해서 시드하지 않음.

## 13. 탐색 경험 구현

와이어프레임(11번)대로 홈 `/`, 탐색 `/explore`, 촬영자 상세 `/photographers/[id]` 구현. 실제 시드 데이터(지호)로 lint/타입체크/빌드/런타임까지 확인 완료.

- **`/`, `/explore`, `/bookings`, `/mypage`, `/favorites`를 `(main)` 라우트 그룹으로 이동**하고 공통 `BottomNavigation`을 레이아웃에 둠 — 4개 탭 모두에서 하단 내비게이션이 일관되게 보이도록. `/photographers/[id]`와 예약 플로우는 그룹 밖에 남겨 하단 내비 대신 자체 CTA를 씀.
- **필터는 URL 쿼리 파라미터(`?view=&purpose=&mood=`) 기반.** 칩을 클라이언트 상태가 아니라 `<Link>`로 구현해 서버 컴포넌트에서 그대로 데이터를 다시 조회 — 불필요한 클라이언트 상태를 만들지 않기 위함.
- **지역·가격 필터, 정렬은 이번 구현에서 뺌.** 와이어프레임엔 있었지만 지역은 고정 목록이 없고 가격은 범위 컨트롤이 필요해 완성도 있게 만들려면 별도 작업이 필요함 — 동작 안 하는 필터 버튼을 보여주지 않기 위해 아예 뺐고, 목적·무드 칩만 실제로 동작하게 구현.
- **찜(♡) 버튼도 이번엔 뺌.** 로그인 확인 + `favorites` 테이블 연동이 필요해 별도 작업 범위로 분리 (RLS는 이미 준비돼 있음).
- **`shooting_services.areas`, `photographer_areas` 등은 이미 있는 컬럼을 그대로 사용**, 새 테이블 추가 없음.
- **촬영자 상세의 "완료 촬영 수"는 0으로 고정.** `bookings` 테이블 RLS가 당사자(고객/촬영자)만 조회를 허용해서 비로그인 방문자는 공개 페이지에서 집계를 낼 수 없음 — 지금은 완료된 예약이 실제로 전무해 0이 정확한 값이지만, 예약 기능이 생기면 완료 건수만 안전하게 노출하는 공개 집계 함수(RPC)가 필요함(TODO로 코드에 남겨둠). 후기 수는 `reviews` 테이블이 이미 전체 공개 RLS라 실제 값을 그대로 집계.

## 14. 예약 플로우 구현

와이어프레임(12번)대로 서비스선택→일정→장소→예약확인→완료→내예약 6개 화면 구현.

- **예약 진행 상태는 Zustand(`localStorage` persist)로 관리.** 명세 22번이 명시한 대로 — 여러 라우트에 걸친 흐름이라 서버 컴포넌트만으로는 유지가 안 됨. 서비스 선택 시 표시용 정보(가격·소요시간·촬영자명 등)까지 함께 저장해 이후 단계에서 재조회 없이 바로 씀.
- **인증 게이트는 예약 플로우 4단계 서버 페이지 각각에 둠** (`requireAuthUser`, `/login?next=`으로 리다이렉트) — 딥링크로 중간 단계에 바로 들어와도 안전하게 막힘.
- **예약 가능 시간 계산은 서버에서.** `availability_rules`(요일별 운영시간) − `blocked_times` − 기존 `bookings`(버퍼 포함)를 30분 단위로 계산하는 순수 함수(`getAvailableSlots`)를 만들고, 일정 선택 화면은 Server Action으로 이 함수를 호출해 날짜를 바꿀 때마다 다시 계산.
- **예약 생성 Server Action은 클라이언트 값을 하나도 신뢰하지 않음.** 서비스 id로 가격·소요시간·버퍼·최대인원을 DB에서 다시 조회해 스냅샷을 만들고, 주말/야간/이동 추가요금도 서버가 재계산. `EXCLUDE` 제약 위반(`23P01`)은 "이미 예약된 시간" 한국어 메시지로 변환.
- **완료 촬영 수는 여전히 0 고정, 후기는 실제 값.** 13번 항목과 동일한 이유(RLS상 공개 집계 불가) — 예약 기능이 추가된 지금도 아직 `completed` 상태 예약이 없어 정확한 값.
- **검증**: lint/타입체크/빌드 통과 + 비로그인 상태에서 4단계 전부 `/login`으로 리다이렉트되는 것 확인 + 테스트 계정으로 실제 REST API를 통해 예약을 생성하고, 겹치는 시간으로 재시도했을 때 `EXCLUDE` 제약이 `23P01`로 정확히 막는 것까지 실제 DB에서 확인.

## 17. 버그: Storage upsert 업로드가 RLS에 막히던 문제

**증상**: 실제 브라우저에서 포트폴리오 사진 업로드 시 "new row violates row-level security policy" 에러로 항상 실패. REST API로 같은 계정·같은 경로에 직접 업로드하면 성공해서 처음엔 원인을 못 찾음 — `@supabase/supabase-js` 클라이언트 라이브러리로 재현 스크립트를 만들어서야 동일 에러를 재현함.

**원인**: 12번 항목에서 Supabase 보안 어드바이저의 "공개 버킷에서 전체 목록 조회 가능" 경고를 없애려고 `avatars`/`portfolios`/`services` 버킷의 SELECT 정책을 전부 삭제했었음. 그런데 `ImageUploader`가 쓰는 `upload(..., { upsert: true })`는 내부적으로 "같은 경로에 파일이 있으면 덮어쓰기" 위해 충돌 여부를 확인해야 하고, 이 확인 과정에 SELECT 권한이 필요함. SELECT 정책이 아예 없으니 업로드(사실상 upsert) 자체가 막힘.

**수정**: 전체 공개 조회가 아니라 **본인 폴더만 조회 가능한 SELECT 정책**을 다시 추가함(`{bucket}_owner_select`). 공개 URL 직접 접근(공개 버킷)과는 무관하고, "전체 목록 조회" 문제도 재발하지 않음(본인 소유 파일만 SELECT 가능) — 실제로 어드바이저 재확인 결과 "목록 조회 가능" 경고는 다시 뜨지 않음.

**교훈**: 어드바이저 경고를 고칠 때 "왜 이 정책이 있었는지"를 다시 따져보지 않고 삭제만 하면, 목록 노출은 막아도 기능 자체가 깨질 수 있음. 이후 스토리지 정책을 바꿀 때는 실제 업로드까지 재현 테스트하기로 함.

## 18. "연락 또는 안내 정보" 필드를 플랫폼 내 안내문으로 재정의

촬영자 등록 3단계 필드(명세 14.13 "연락 또는 안내 정보")의 placeholder가 "카카오톡 오픈채팅 링크 또는 이메일"이었는데, 이는 3번 섹션이 해결하려는 문제("오픈채팅이나 개인 거래는 신뢰하기 어렵다")와 정면으로 배치됨 — 사용자를 플랫폼 밖 채널로 유도하는 셈이라 지적받고 수정함.

- **라벨/문구를 "촬영 안내사항"으로 바꾸고, 외부 연락처 대신 예약 진행에 필요한 안내(준비물, 만남 방법 등)를 적도록 힌트 문구 추가.** DB 컬럼명(`contact_info`)은 그대로 두되(마이그레이션 비용 대비 실익 낮음), 사용자에게 보이는 라벨과 예시만 바꿈.
- **이 필드를 예약 완료 화면(14.9)의 "준비 사항"에 실제로 연결함.** 이전에는 하드코딩된 문구만 보여줬는데, 이제 촬영자가 등록한 안내문이 있으면 그걸 보여주고 없으면 기본 문구로 대체 — 필드가 실제로 플랫폼 안에서 쓰이는 목적을 갖게 됨.

## 19. 촬영자 예약함 구현

15번 섹션 "예약 승인 및 거절은 핵심 흐름 검증에 필요하다고 판단되면 단순 상태 변경 형태로 구현할 수 있습니다"에 따라 구현.

- **탭 4개(요청중/예정/완료/취소·거절)**, 고객 쪽 `/bookings`와 동일한 패턴이지만 상대방 표시가 촬영자 이름이 아니라 **고객 닉네임**임 — `profiles`는 RLS상 본인만 조회 가능해서, 공개용 `profiles_public` 뷰를 통해 조회(같은 FK에 뷰가 대체 조인 대상으로 잡혀 있어 `profiles_public!bookings_customer_id_fkey(nickname)` 형태로 조회 가능했음).
- **상태 전이는 Server Action에서 명시적으로 검증**: `requested→confirmed/rejected`, `confirmed→completed`만 허용. RLS는 "이 예약의 당사자인가"만 검사하고 "어떤 상태에서 어떤 상태로 바뀔 수 있는가"는 검사하지 않으므로(4번 항목 결정과 동일한 이유), 앱 레이어에서 현재 상태를 재확인 후 업데이트.
- **"촬영 완료로 표시" 버튼은 촬영 일시가 지난 예약에만 노출.** 아직 촬영 전인데 완료 처리되는 걸 방지.
- **검증**: lint/타입체크/빌드 통과. 테스트 계정으로 REST API를 통해 실제 예약을 `requested→confirmed`로 전이시키고, `booking_status_logs`에 상태 변경 이력이 정확한 행위자(customer_id로 생성, photographer_id로 승인)와 함께 자동 기록되는 것까지 DB에서 확인. 브라우저에서 바로 승인/거절 버튼을 테스트할 수 있도록 테스트 촬영자 계정에 "요청중" 예약 1건을 추가로 만들어둠.

## 20. 모바일 UI 검수 및 수정

명세 17.4(모바일 기준 체크리스트) 대로 지금까지 만든 화면 전체를 정적 코드 리뷰로 훑음(하단 고정 내비/CTA 겹침, safe-area, 가로 스크롤, 키보드 가림, 텍스트 오버플로, 터치 영역 7개 항목). 하단 내비·safe-area 겹침 문제는 없었고, 아래 두 종류만 실제 수정함.

- **터치 영역**: 44px는 기계적 기준이 아니라 참고치로 다루기로 함(사용자 지시). 오작동 시 대가가 크거나(뒤로가기가 패딩 없이 아이콘만 있던 경우, 승인/거절처럼 인접한 반대 동작) 실제로 탭 영역이 사실상 없던 경우만 넓힘 — `step-header.tsx`/`onboarding-wizard.tsx`의 패딩 없는 뒤로가기(`-ml-2 size-11`로), 이미지 삭제 X(`size-5`→`size-7`), 촬영자 예약함의 승인/거절/완료 버튼(`min-h-11`). 반대로 라이트박스 닫기/이전·다음(이미 36~40px, 화면 모서리에 독립 배치), 서비스 상세·홈의 원형 뒤로가기/검색 아이콘(이미 32px 원형 배경 확보, 주변에 경쟁 요소 없음), 태그 입력의 "추가" 버튼(flex stretch로 이미 ~36px)은 이미 충분하다고 판단해 그대로 둠. "이 서비스로 예약" 칩은 44px로 키우면 카드 안에서 시각적으로 과하게 커져서 `py-1→py-2` 정도로만 절충.
- **긴 텍스트 오버플로**: 활동명·헤드라인·서비스명·닉네임처럼 사용자가 입력하는 텍스트가 flex 행 안에 있으면 `min-w-0` 없이는 잘리지 않고 흘러넘칠 수 있어 `min-w-0 truncate`(한 줄, 카드형)를, 중앙 정렬된 제목류에는 `wrap-break-word`(공백 없는 긴 한글 문자열 대비)를 추가함 — `photographer-list-item.tsx`, `photographers/[id]/page.tsx`(이름/헤드라인/서비스명), `photographers/[id]/services/[serviceId]/page.tsx`(작가 닉네임 행/제목), `mypage/page.tsx`(닉네임).
- **하단 여백**: 예약 플로우 4단계 중 `confirm`만 `pb-28`이고 나머지 3단계(`service`/`schedule`/`location`)는 `pb-24`였음. 고정 CTA 바 마크업이 4단계 모두 동일한데 패딩만 달랐던 것이라 `pb-28`로 통일 — 노치 기기에서 safe-area가 큰 경우 콘텐츠 마지막 줄이 CTA 바에 살짝 가릴 여지를 없앰.
- **검증**: lint/타입체크/프로덕션 빌드 통과 확인.

## 21. 홈 화면 "지금 활동 중인 촬영자" 스토리 바 추가

인스타그램 무드로 "둘러보기"(홈/탐색/촬영자 상세) 화면을 개선하는 논의의 일부로, 홈 상단에 스토리형 촬영자 아바타 바를 추가함. 와이어프레임([아티팩트](https://claude.ai/code/artifact/a697a76b-7d00-4da3-aba7-b3df90e67295))으로 먼저 검토 후 구현.

## 22. 브랜드명 확정 및 로고 적용

브랜드명·컬러 논의(§21 이후)를 거쳐 "무디"(뉴트럴/블랙 톤) 방향으로 좁혀지다가, 최종적으로 영문 단독 표기 "moodi"(소문자)로 확정. 사용자가 직접 제작한 워드마크 SVG(`moodi_wordmark_v1.svg`)를 그대로 로고로 채택.

- **적용 위치**: `public/logo.svg`로 저장 후 홈 헤더([src/app/(main)/page.tsx](../src/app/(main)/page.tsx))의 텍스트 로고를 이 SVG로 교체, `next/image`로 렌더링(SVG는 자동으로 `unoptimized` 처리되어 `next.config.ts`에 `dangerouslyAllowSVG` 설정이 필요 없음 — Next 16 이미지 문서 확인함). 브라우저 탭 타이틀(`src/app/layout.tsx`)도 "moodi — 사진 촬영 예약"으로 변경.
- **폰트를 Suisse Int'l(유료) → Inter(오픈소스)로 교체**(`moodi_wordmark_openfont.svg`, 사용자가 재작업해 전달). Inter도 OS 기본 내장 폰트는 아니라서 `<img>`로 로드되는 이 SVG가 뷰어 기기에 Inter가 없으면 여전히 `Arial/Helvetica` 폴백으로 렌더링될 수 있다는 점은 동일하게 남아 있음(외부 SVG 파일은 `<img>`로 불러오면 문서 자체 폰트 컨텍스트가 격리되어, 페이지에서 `next/font`로 Inter를 로드해도 이 SVG에는 적용되지 않음) — 사용자가 "일단"이라는 표현으로 임시본임을 밝혀 그대로 반영. 완전히 해결하려면 텍스트를 패스로 아웃라인 변환하거나 SVG 내부에 폰트를 `@font-face`로 직접 임베드해야 함.
- **정사각형 파비콘·앱 아이콘 추가**: 워드마크가 1400×420 가로형이라 정사각형 자리에 못 써서, `src/app/icon.svg`(정적 SVG, 검정 배경 + 흰색 소문자 "m", Next 16 파일 컨벤션으로 자동 `<link rel="icon">` 주입됨 — `favicon.ico`/수동 `<link>` 불필요, [Next 16 아이콘 문서](../node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/01-metadata/app-icons.md) 확인함)와 `src/app/apple-icon.tsx`(`next/og`의 `ImageResponse`로 PNG 코드 생성, apple-icon은 svg를 지원하지 않아 png만 가능 — Satori는 시스템 폰트 접근이 없어 커스텀 폰트를 지정해도 실제 폰트 바이너리를 `fonts` 옵션으로 넘기지 않으면 무시되므로, 폰트 지정 없이 기본 폴백 굵은 글씨로 둠)를 추가.

## 24. 이메일 인증(Confirm email) 끔

배포 전 실사용자 검증을 앞두고, Supabase 기본 SMTP가 조직 소속 이메일에만 발송 가능하다는 제약을 정면 돌파하는 대신 우회하기로 함. 이 앱엔 아직 비밀번호 재설정 기능이 없어서 이메일 발송에 의존하는 흐름이 회원가입 확인 하나뿐이었고, 결제·민감정보 접근 같은 악용 가능 기능도 없어서 이메일 미인증 리스크가 낮다고 판단.

- 사용자가 Supabase 대시보드(Authentication → Providers → Email)에서 "Confirm email" 토글을 직접 끔.
- `src/app/signup/actions.ts`의 `signUpAction`은 이미 `data.session` 존재 여부로 분기하도록 짜여 있어서(이메일 인증 켜짐/꺼짐 양쪽 다 대응) 코드 변경 없이 그대로 동작.
- **검증**: 실제 `@supabase/supabase-js` 클라이언트로 새 테스트 계정을 가입시켜 (1) `signUp` 응답에 세션이 즉시 포함되는지, (2) `email_confirmed_at`이 즉시 채워지는지, (3) 가입 직후 `signInWithPassword`로 정상 로그인되는지 확인 — 셋 다 통과. 테스트 계정은 `service_role`로 삭제해 정리함.
- **재검토 시점**: 비밀번호 재설정 기능을 추가하거나, 정식 공개 런칭 시 이메일 인증을 다시 요구하고 싶을 때 — 그때는 커스텀 SMTP(Resend/SendGrid 등) 연결이 다시 필요함.

## 26. git 초기화 및 GitHub 연결

배포를 위해 처음으로 git 저장소화. Public 저장소 사용, 사용자가 미리 만들어둔 `https://github.com/EunGyeoll/PicMatching`에 연결.

- **비밀키 유출 확인**: `git add -A` 전에 `.gitignore`에 `.env*`(단 `.env.example` 제외)가 이미 포함돼 있는 걸 확인했고, 스테이징 결과에도 `.env.local`이 없는 것을 재확인함. `.env.example`은 실제 값 없이 빈 템플릿이라 커밋해도 안전. `SUPABASE_SERVICE_ROLE_KEY`가 Public 저장소에 올라가면 안 되므로 특히 신경 씀.
- **원격 저장소가 완전히 비어있지 않았음**: 처음 push가 거부됨 — GitHub에서 저장소 생성 시 기본으로 추가되는 `README.md`(내용 "# PicMatching") 한 줄이 이미 있었음. `git pull --rebase`로 받아온 뒤 `README.md` 충돌만 발생(둘 다 새 파일로 추가해서 add/add 충돌) — 로컬의 더 완성된 README를 유지하되 제목만 "moodi"로 맞춰서 해결.
- **커밋 작성자**: 전역 git 설정에 이미 `user.name`/`user.email`이 잡혀 있어 별도 설정 불필요.
- **다음 단계**: Vercel 연동은 이 환경에 CLI(`gh`, `vercel`)가 없고 계정 인증도 대신할 수 없어 사용자가 직접 진행.

## 28. 배포 후 실사용 중 발견된 placeholder 화면 수정 — 서비스 목록

배포된 사이트에서 실제로 촬영자 등록 → 서비스 등록을 시도하던 중 발견. `/photographer/services`(nav의 "서비스" 탭이자 서비스 등록 완료 후 리다이렉트 대상)가 스캐폴딩 단계의 `RoutePlaceholder`("촬영 서비스 관리 화면 준비 중입니다")로 남아있었음 — `services/new`(등록 폼)는 실제로 동작했지만, 그 진입점이자 등록 후 도착지인 목록 화면이 막혀 있어 사실상 서비스 등록 기능 전체가 막힌 것과 같았음.

- **`getMyServices(photographerId)`** (`src/lib/data/services.ts`) 추가 — 공개 여부와 무관하게 본인 서비스 전체를 반환(기존 `getFeaturedServices`/`getExploreServices`는 `is_published=true`만 노출하는 공개용이라 재사용 불가).
- **`src/app/photographer/(dashboard)/services/page.tsx`**를 placeholder에서 실제 목록 화면으로 교체 — 커버 이미지, 제목, 가격/시간, 공개·비공개 배지, 상단에 "새 서비스 등록" 링크. 삭제·수정 액션은 이번 범위에 포함 안 함(수정 화면 `[id]/edit`은 여전히 placeholder — 아래 열린 질문에 남김).
- **검증**: lint·타입체크·프로덕션 빌드 통과. 비로그인 상태로 해당 경로 요청 시 크래시 없이 로그인 페이지로 정상 리다이렉트되는 것 확인. DB에서 테스트 계정(test-photographer, 발행 서비스 2건)으로 실제 반환될 데이터 형태를 사전에 SQL로 대조.
- **별개로 발견한 사소한 기존 이슈**: `(dashboard)/layout.tsx`의 인증 가드가 `requireAuthUser("/photographer/dashboard")`로 next 경로가 고정돼 있어서, 세션 만료 상태로 어느 탭에 있었든 재로그인 후 항상 대시보드로 돌아감(원래 있던 탭으로 안 돌아옴) — 이번 수정 범위 밖이라 손대지 않음.

## 30. 서비스 수정 · 삭제 · 공개 전환 구현

§28에서 목록 화면만 살려뒀던 것에 이어, 등록한 서비스를 실제로 관리(수정/삭제/공개 전환)할 수 있게 함.

- **`ServiceForm`을 등록/수정 공용 컴포넌트로 리팩터링**(`services/new/service-form.tsx` → `services/service-form.tsx`로 이동). `mode: "create" | "edit"` + `initial?: ServiceEditData`로 분기 — edit일 때 `serviceId`를 새로 생성하지 않고 기존 id를 그대로 쓰고, 모든 필드에 `defaultValue`/`defaultChecked`로 기존 값을 채움.
- **`updateServiceAction`** 추가 — `createServiceAction`과 동일한 검증(zod 스키마 재검증)을 거치되, 본인 소유(`photographer_id` 일치) 확인 후 UPDATE. 태그는 전체 삭제 후 재삽입(`syncServiceTags`, 온보딩 RPC의 `photographer_areas` full-replace 패턴과 동일 방식) — INSERT/UPDATE 중복 로직은 `toRow()` 헬퍼로 통합.
- **`getServiceForEdit(photographerId, serviceId)`**(`src/lib/data/services.ts`) 추가 — 본인 소유가 아니면 `null` 반환, 수정 화면에서 `notFound()` 처리.
- **`deleteServiceAction`**: `bookings.service_id → shooting_services.id`가 `ON DELETE RESTRICT`로 걸려 있어(예약 이력 보호 목적으로 이미 그렇게 설계돼 있었음) 예약 이력이 있는 서비스는 하드 삭제가 DB 레벨에서 막힘. Postgres 에러 코드 `23503`을 캐치해 "예약 이력이 있으면 비공개로 전환해달라"는 한국어 안내로 변환.
- **`togglePublishAction`**: `is_published`만 바꾸는 가벼운 액션. 예약 이력이 있어 삭제가 막히는 서비스의 사실상 "삭제" 대안.
- **`ServiceRowActions`**(client) — 목록의 각 서비스 행에 수정/공개·비공개 전환/삭제 텍스트 링크. 삭제는 `window.confirm()`으로 실수 방지.
- **덤으로 발견해 고친 버그**: 기존 등록 폼에 "기타 안내사항"(`notes`) 입력란이 있었는데 `createServiceAction`의 insert 객체에 `notes` 필드가 빠져 있어서 사용자가 입력해도 저장되지 않고 조용히 사라지고 있었음(DB 컬럼은 이미 존재). `toRow()`로 통합하면서 같이 수정.
- **검증**: lint·타입체크·프로덕션 빌드 통과. 실제 테스트 계정(test-photographer)으로 `@supabase/supabase-js` 클라이언트를 통해 (1) 공개 여부 토글 양방향 성공, (2) 예약 이력이 있는 서비스 삭제 시도 시 정확히 `23503`으로 막히는 것, (3) 예약 이력이 없는 임시 서비스는 생성 후 삭제까지 정상 동작하고 실제로 행이 사라지는 것까지 DB에서 직접 확인. 테스트로 만든 임시 서비스는 스스로 삭제되어 잔여 데이터 없음.

## 31. 일반 사용자 프로필 사진 기능 추가

촬영자는 온보딩 1단계에서 아바타를 올릴 수 있었지만, 일반 사용자(§6.6 — 모든 회원은 우선 일반 사용자로 가입)는 마이페이지에서 닉네임·연락처만 수정 가능하고 프로필 사진을 넣을 방법이 아예 없었음. `profiles.avatar_url` 컬럼과 `avatars` 버킷은 이미 있었으므로 온보딩과 같은 방식으로 노출만 추가.

- **`mypage/edit`**: `ImageUploader`(bucket=`avatars`, shape=`circle`, 온보딩과 동일하게 `${userId}/profile.<ext>` 경로) 추가. 저장은 hidden input으로 전체 공개 URL을 넘기고 `updateProfileAction`이 `profiles.avatar_url`에 그대로 저장 — `avatar_url` 컬럼 자체가 storage 상대 경로가 아니라 완성된 URL을 저장하는 기존 관례(온보딩 때부터)를 그대로 따름.
- **기존 아바타를 수정 화면에 다시 채우는 문제**: `ImageUploader`는 `{path, url}` 형태를 기대하는데 DB엔 URL만 있어서, `getPublicStorageUrl`이 만드는 URL 접두사(`.../storage/v1/object/public/avatars/`)를 떼어내 path를 역산하는 방식으로 해결(`mypage/edit/page.tsx`). 이 방식이 실제로 원래 path와 정확히 일치하는지 아래 검증에서 문자열 그대로 대조함.
- **마이페이지 본문에도 아바타 표시**: 이전엔 `avatar_url`을 이미 select하고 있었는데 화면에 그리지는 않고 있었음 — 닉네임 옆에 원형 이미지(없으면 닉네임 첫 글자) 추가.
- **검증**: lint·타입체크·프로덕션 빌드 통과. 테스트 계정(test-user)으로 실제 PNG를 생성해 `avatars` 버킷에 업로드 → `profiles.avatar_url` 갱신 → URL에서 path를 역산한 값이 원래 업로드 경로와 정확히 일치 → 실제 공개 URL이 200으로 이미지를 서빙하는 것까지 확인. 테스트 후 스토리지 파일 삭제 + `avatar_url`을 다시 `null`로 되돌려 공유 테스트 계정에 잔여 데이터를 남기지 않음.

## 32. 대시보드에 "촬영자 정보" 진입점 추가

촬영자 활동명·소개·연락처(§31에서 설명한, `photographer_profiles`의 닉네임과는 별개인 필드들)를 고칠 방법이 UI 어디에도 없다는 걸 사용자가 지적해 발견. `/photographer/onboarding` 페이지 자체는 이미 기존 값을 채워서 열리는 등록/수정 겸용으로 짜여 있었지만(§16), 대시보드 nav 어디에도 연결이 안 되어 있어 URL을 직접 아는 사람만 접근 가능했음.

- `(dashboard)/layout.tsx`의 nav에 "촬영자 정보" 항목 추가, `/photographer/onboarding`로 연결.
- **이탈 동선도 같이 정리**: `onboarding/`은 `(dashboard)` 라우트 그룹 밖에 있어서(첫 등록 전 사용자에게 촬영자 nav가 먼저 안 보이게 하려는 의도적 설계, §16) 대시보드 nav 없이 온보딩 화면만 뜨는데, 1단계엔 뒤로가기 버튼이 아예 없었음. 1단계에서도 뒤로가기 버튼을 보이게 하고, `router.back()`으로 원래 있던 곳(마이페이지든 대시보드든)으로 돌아가게 함.
- **수정 모드 문구 분기**: 기존 활동명이 있으면(`initialDisplayName` 존재) 편집 중이라고 보고 헤더("촬영자 등록"→"촬영자 정보 수정"), 마지막 단계 버튼("등록 완료"→"수정 완료"), 안내 문구("완료 후 마이페이지에서 촬영자 모드로 전환할 수 있어요"→"수정한 내용은 촬영자 상세 화면에 바로 반영돼요")를 모두 상황에 맞게 바꿈. 신규 등록 문구 그대로 뜨면 이미 촬영자인 사람이 다시 "등록"하는 것처럼 보여 혼란스러울 것 같았음.
- **검증**: lint·타입체크·프로덕션 빌드 통과.

## 33. 버그: 온보딩 재제출 시 포트폴리오 사진 중복 저장

사용자가 실사용 중 발견: 촬영자 포트폴리오에 같은 사진이 두 번씩 저장돼 있음. §32에서 대시보드에 "촬영자 정보" 편집 진입점을 추가하면서 실제로 드러난 버그 — `complete_photographer_onboarding` RPC(§16)가 `photographer_areas`는 `delete` 후 `insert`(멱등)로 처리하면서 `portfolio_images`는 매번 그냥 `insert`만 하고 있었음. 첫 등록 때는 한 번만 호출돼서 문제가 없었지만, 이제 온보딩 화면을 편집용으로 재방문·재제출할 수 있게 되면서 재제출할 때마다 기존 사진이 중복 삽입되는 게 실제로 발생함.

- **수정**: RPC에 `delete from portfolio_images where photographer_id = v_uid;`를 insert 앞에 추가 — `photographer_areas`와 동일한 delete-then-insert 패턴으로 통일(마이그레이션 `fix_onboarding_portfolio_idempotency`). `portfolio_images.id`를 참조하는 FK가 없는 것을 먼저 확인하고 진행.
- **기존 중복 데이터 정리**: 영향받은 계정(test-user, `잘찌금`)의 중복 행 2건을 직접 삭제. 정리 후 이 계정 포트폴리오는 3장 → 2장이 됨 — 재제출 시점에 사용자가 위자드에서 사진 1장을 실제로 지우고 저장했을 가능성이 있어(의도적 삭제라면 정상 동작), 사용자에게 확인 필요.
- **검증**: 실제 계정으로 동일한 파라미터(현재 DB 값 그대로)로 RPC를 연속 2번 호출해 포트폴리오 개수가 늘어나지 않는 것 확인(2→2→2). `sort_order`도 매 호출마다 제출한 배열 순서로 다시 정규화됨(기존엔 위치가 안 맞을 수 있었음).

## 34. 마이그레이션을 `supabase/migrations/`에 파일로도 저장하기 시작

§33 버그 대응 중 "DB 함수 수정이 git엔 전혀 안 남는다"는 지적을 받고, 지금까지 모든 스키마·함수 변경을 MCP `apply_migration`으로 라이브 DB에만 적용하고 이 로그에 서술로만 남겨온 방식의 한계를 확인함.

- `supabase/migrations/`에 지금부터 적용하는 SQL을 파일로도 저장. 첫 파일은 §33의 수정 SQL(`20260718081322_fix_onboarding_portfolio_idempotency.sql`).
- **소급 복원은 하지 않음** — 이전 변경들은 이 폴더에 없고, 계속 이 문서(§1~33)가 유일한 기록. `supabase/migrations/README.md`에 이 경계를 명시해둠.
- Supabase CLI(`supabase db push`/`pull`)는 아직 이 환경에 없어서 로컬 재생 워크플로까지는 안 갖춰짐 — 지금은 이력 추적·리뷰 목적으로만 파일을 남기는 단계.

## 35. 아직 결정하지 않은 것 (열린 질문)

- 로컬/배포 Node.js 버전을 22로 올릴지 여부.
- 커스텀 SMTP 연결 — 비밀번호 재설정 기능 추가 또는 정식 런칭 시 필요(§24).

- **아직 placeholder로 남아있는 화면들**: `/photographer/dashboard`(대시보드 개요), `/photographer/availability`(일정). 실사용 중 필요해지는 대로 처리.
